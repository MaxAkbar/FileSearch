Workflows

Save multi-step searches and replay them with one click.

A workflow is a named sequence of steps: run a search, branch on how much it found, narrow into the matched files, loop with broader parameters, then export the results, copy the files, or hand them to a program. Open the editor with Search > Workflows... or Workflows in the sidebar.

Step Types

Step What it does
Search Runs a query over root folders, or — with a scope step set — only over the files an earlier search matched (a sub-search). Filters include file-name globs (matched against the name only, never the path), size and date limits, and a list of directory names to skip (.git, .vs and node_modules by default). Results are recorded under the step's id for later steps.
If Evaluates a condition over earlier search results and runs the then steps when it holds, otherwise the else steps.
Retry loop Repeats its body until a condition is met or the iteration limit is reached — the "broaden and try again" loop. Optional parameter sets give each iteration its own variables.
For each file Runs its body once per distinct file a search matched, exposing the current file to the body as variables.
Export results Writes search results to a JSON, CSV, or Markdown file, asking for confirmation before an existing file is overwritten. New files are written without a prompt.
Copy / move files Copies or moves the matched files into a folder, with a collision policy (rename, skip, or overwrite).
Run program Launches an external program once, or once per matched file with the file path available in the arguments. Write "${file}" in quotes — file names may contain spaces. When waiting for exit, a non-zero exit code or a timeout marks the step failed; untick Wait for exit for GUI apps you don't want to wait on.
Stop Ends the workflow immediately with a success or failure and an optional message — typically inside an if-branch.

Conditions

If-steps and retry loops compare a number from an earlier search step: the hit count (matched lines) or the file count (distinct files with at least one hit). Choose the source step by id, or leave it at (last search) to measure the most recent search. For example: hitCount of 'find-todos' >= 5.

Every step has a unique id (generated automatically, e.g. search-1). Conditions, sub-searches, exports and file operations reference results by these ids, so the editor offers a dropdown of the search steps defined so far. A reference must point at a search step that runs before the referencing step; the one exception is a retry loop's exit condition, which may reference a search inside the loop's own body.

Loops and Variables

Steps inside loop bodies can use ${variable} placeholders in the query text, root folders, globs, export paths, destination folders, and program arguments:

Variable Available in Value
${file} For-each bodies, per-file program arguments Full path of the current file.
${fileName} For-each bodies, per-file program arguments File name without the folder.
${directory} For-each bodies, per-file program arguments Folder containing the current file.
${iteration} Retry bodies The 1-based iteration number.
Parameter-set keys Retry bodies Iteration i exposes the keys of the i-th parameter set, so successive iterations can relax the query, widen roots, and so on.

Parameter sets are edited as key=value lines, one set per iteration; values cannot span lines, so hand-edit the JSON file for multi-line values. The built-in names — file, fileName, directory, iteration — are reserved: a parameter-set key with the same name (case-insensitive) overrides the built-in for that iteration. Unknown ${name} placeholders are left as-is.

Running a Workflow

  • Run executes the workflow. The run log shows each step starting and completing (indented by nesting), and the hits list streams matching lines as they are found. The display is capped at the first 5,000 hits; counting continues past the cap.
  • Dry run performs the searches but no exports, file operations, or program launches — those steps log what they would do instead. Searches still read your files, so you can check hit counts before a real run.
  • Cancel stops the run; closing the window also cancels it.
  • Before any files are copied or moved, any program is launched, or an export overwrites an existing file, FileSearch shows a confirmation listing what is about to happen. Declining skips that step and the workflow continues. Exports to a new file path write without a prompt.
The editor validates as you type — missing names, empty loops, duplicate ids, more parameter sets than iterations, and references to search steps that don't run earlier are listed under the step editor — and a workflow must be valid before it runs.

Where Workflows Are Stored

Each workflow is one JSON file in %AppData%\FileSearch\Workflows in a documented, hand-editable format (see README.Workflows.md in the project repository). Use Folder in the workflow library to open it. To share a workflow, send the file; to install one, drop it into the folder and reopen the window. Files that fail to parse still appear in the library with their error message.

A workflow file is executable automation — it can copy or move files, overwrite export files, and launch programs. Review the steps before running a workflow someone else wrote. Confirmations cover file operations, program launches and export overwrites, but an export to a new path writes without prompting.
